# Redis Migration and Edge Auth Analysis & Implementation Roadmap

**Project:** TheGame Admin Web-App
**Date:** March 10, 2026
**Status:** Analysis & Planning Phase
**Sprint:** Phase 13.4 - Monthly Close Workflow and Database Migration

---

## Executive Summary

This document outlines a critical infrastructure refactoring to implement **strict database namespacing**, **safe data migration**, and **100% Edge-safe authentication** for TheGame admin web-app. Currently, TheGame shares a single Upstash Redis database with two other projects ("Pixelbrain" and "akiles-ecosystem"), creating potential data collisions and bandwidth inefficiencies.

### Current State Analysis

**Critical Issues Identified:**

1. **No Database Isolation:** All projects share the same Upstash Redis keys without namespacing
   - Current key patterns: `data:${entity}:${id}`, `account:${identifier}`, `index:${entity}`, `archive:*`, `effects:*`, `logs:*`
   - Risk: Key collisions across projects, data integrity issues, debugging complexity

2. **Authentication Not Edge-Safe:** Every authentication request causes Redis roundtrips
   - `AuthService.verifySession()` makes 2-3 Redis calls per middleware hit: `kvGet<Account>`, `kvGet<Character>` to verify sessions
   - This prevents true Edge runtime deployment and consumes excessive bandwidth
   - Shared database makes this even more critical

3. **Missing Rate Limiting:** No protection against API abuse or bandwidth exhaustion
   - Heavy API routes (login, data operations) have no rate limiting
   - Upstash Redis bandwidth costs could escalate rapidly

4. **Static Configuration Loading:** Repeated Redis access for immutable data
   - `PERMISSION_MATRIX` and other configs are loaded from Redis on every request
   - Should use `unstable_cache` for read-heavy, static data

## Phase 1: Database Migration Strategy

### 1.1 Migration Script Design

**Objective:** Safely rename all existing global keys to use `thegame:` namespace prefix.

**Key Requirements:**
- Zero data loss guarantee
- Atomic operations using Redis RENAME
- Pagination with SCAN to avoid timeout
- Comprehensive logging and rollback capability
- Handle all key patterns currently in use

**Current Key Patterns to Migrate:**
```
data:*                → thegame:data:*
account:*             → thegame:account:*
index:*               → thegame:index:*
archive:*             → thegame:archive:*
effects:*             → thegame:effects:*
links:*               → thegame:links:*
logs:*                → thegame:logs:*
data:player-*         → thegame:data:player-*
data:company-*        → thegame:data:company-*
```

**Migration Script Specifications:**

```typescript
// Key Features:
// - Uses Redis SCAN with COUNT 1000 for efficient iteration
// - RENAME is atomic - no data loss if interrupted
// - Batch processing with progress tracking
// - Validates key patterns before migration
// - Generates comprehensive audit log
// - Supports dry-run mode for testing
// - Includes rollback capability
```

**Execution Strategy:**
1. **Backup Phase:** Export all current keys to a timestamped JSON backup
2. **Dry Run:** Execute script with dry-run flag to simulate migration
3. **Validation:** Verify dry-run output shows correct transformations
4. **Live Migration:** Execute live migration during low-traffic window
5. **Post-Migration Verification:** Sample keys to confirm successful rename
6. **Rollback Plan:** Keep backup available for 7 days

**Estimated Migration Time:** 2-5 minutes for 1,000-10,000 keys (typical range)

**Rollback Procedure:**
- If issues detected, execute reverse migration script
- RENAME all `thegame:*` keys back to original patterns
- Restore from backup if atomic rename fails

## Phase 3: Stateless Edge Auth Refactor

### 3.1 Current Authentication Bottleneck

**Problem:** `AuthService.verifySession()` makes multiple Redis calls per request:

```typescript
// Current flow (NOT Edge-safe):
static async verifySession(token: string): Promise<AuthUser | null> {
  // 1. Verify JWT signature (local, fast)
  const verified = await verifyJwt(token, secret);

  // 2. Redis Call #1: Fetch account from KV
  const user = await kvGet<Account>(buildAccountKey(userId));

  // 3. Redis Call #2: Fetch character for roles
  const character = await kvGet<Character>(buildDataKey('character', characterId));

  // 4. Redis Call #3 (optional): Fetch additional data

  // ❌ 2-3 Redis roundtrips per authentication check!
  // ❌ Cannot run in Edge runtime due to Redis dependencies
  // ❌ Wastes bandwidth on every API request
}
```

**Impact Analysis:**
- Assume 10 API requests per minute per active user
- Each request: 2-3 Redis calls = ~500 bytes per request
- With 10 users: 5,000 bytes/min = 7.2 MB/day = 216 MB/month just for auth
- Multiply by 3 projects sharing the database = **648 MB/month overhead**

### 3.2 Stateful → Stateless Migration

**Solution:** Pack all necessary user data into JWT payload during login.

**New Authentication Flow:**

```typescript
// Login (Still uses Redis for verification):
static async login(username: string, password: string, rememberMe: boolean): Promise<AuthSession> {
  // 1. Verify credentials (Redis calls required)
  const user = await kvGet<Account>(buildAccountKey(username.toLowerCase()));
  const passwordMatch = await bcrypt.compare(password, user.passwordHash);

  // 2. Fetch user data (Redis calls required)
  const character = await kvGet<Character>(buildDataKey('character', characterId));

  // 3. ✅ PACK EVERYTHING INTO JWT
  const token = await generateJwt({
    sub: user.id,
    email: user.email,
    username: user.name,
    roles: character.roles,              // ← User roles
    characterId: characterId,            // ← Character reference
    isAdmin: character.roles.includes('founder') || character.roles.includes('admin'),
    // Add any other data needed for auth decisions
    permissions: user.permissions || []   // ← Pre-computed permissions
  }, secret, expiresIn);
}

// Middleware (100% Edge-safe, no Redis calls):
export async function middleware(request: NextRequest) {
  const token = request.cookies.get('auth_session')?.value;

  // ✅ Single cryptographic verification only
  const verified = await jwtVerify(token, secret);

  // ✅ Read roles directly from JWT payload
  const userRoles = verified.payload.roles;
  const isAdmin = verified.payload.isAdmin;

  // ✅ No Redis calls, runs in Edge runtime
  return NextResponse.next();
}
```

**Benefits:**
- **0 Redis calls** in middleware for authenticated requests
- **True Edge runtime compatibility**
- **Bandwidth savings:** ~90% reduction in auth-related Redis traffic
- **Performance:** JWT verification is ~1ms vs ~50ms for Redis roundtrip

## Phase 4: Bandwidth Protection Implementation

### 4.1 Rate Limiting with @upstash/ratelimit

**Installation:**
```bash
npm install @upstash/ratelimit
```

**Implementation Strategy:**

1. **Authentication Rate Limiting**
   - Limit login attempts: 5 per 15 minutes per IP
   - Prevent credential stuffing attacks
   - Protect shared database from abuse

2. **API Rate Limiting**
   - General API: 100 requests per minute per user
   - Heavy operations (bulk imports): 10 per hour
   - Read operations: 200 per minute
   - Write operations: 50 per minute

3. **Rate Limiting by Tiers:**
   ```typescript
   // Define rate limits for different operation types:
   const RATE_LIMITS = {
     AUTH: { limit: 5, window: '15 m' },
     API_READ: { limit: 200, window: '1 m' },
     API_WRITE: { limit: 50, window: '1 m' },
     BULK_OPERATION: { limit: 10, window: '1 h' },
   };
   ```

**Rate Limiter Implementation:**

```typescript
// middleware.ts with rate limiting
import { Ratelimit } from '@upstash/ratelimit';
import { Redis } from '@upstash/redis';

const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(5, '15 m'),
  analytics: true,
});

export async function middleware(request: NextRequest) {
  // Rate limiting for auth endpoints
  if (request.nextUrl.pathname.startsWith('/api/auth')) {
    const identifier = request.ip || 'anonymous';
    const { success, limit, remaining, reset } = await ratelimit.limit(identifier);

    if (!success) {
      return NextResponse.json(
        { error: 'Rate limit exceeded', reset },
        { status: 429 }
      );
    }

    response.headers.set('X-RateLimit-Limit', limit.toString());
    response.headers.set('X-RateLimit-Remaining', remaining.toString());
    response.headers.set('X-RateLimit-Reset', reset.toString());
  }

  // ... rest of middleware
}
```

### 4.2 Static Configuration Caching

**Problem:** `PERMISSION_MATRIX` and other static configs are loaded on every request.

**Solution:** Use Next.js 14's `unstable_cache` for static data.

**Implementation:**

```typescript
// lib/auth-service.ts
import { unstable_cache } from 'next/cache';

// Cache PERMISSION_MATRIX for 1 hour (static, rarely changes)
export const getPermissionMatrix = unstable_cache(
  async () => PERMISSION_MATRIX,
  ['permission-matrix'],
  { revalidate: 3600 } // 1 hour
);

// Usage in AuthService:
static getPermissions(user: AuthUser): AuthPermissions {
  const matrix = getPermissionMatrix(); // ← Cached, no Redis call

  // ... permission logic
}
```

**Cache Targets:**
- `PERMISSION_MATRIX` (role-based access control)
- `BUSINESS_STRUCTURE` (station definitions)
- `LOCATION_STRUCTURE` (geographic data)
- Financial conversion rates
- Player conversion rates
- Company/personal assets (cached for 5 minutes)

**Cache Invalidation Strategy:**
- Use `revalidate()` when configs are updated via admin UI
- Time-based revalidation for rarely-changing data
- Manual cache flush during deployments


- `links:*`: ~100 keys (entity relationships)
- **Total: ~1,100 keys**

### B. Auth Flow Comparison

**Before (Stateful):**
```
1. Login: 3 Redis calls (verify password, fetch user, fetch character)
2. Middleware: 2-3 Redis calls (verify session, fetch user, fetch permissions)
3. Total per request: 5-6 Redis calls
```

**After (Stateless):**
```
1. Login: 3 Redis calls (verify password, fetch user, fetch character)
2. Middleware: 0 Redis calls (verify JWT signature only)
3. Total per request: 3 Redis calls (login) / 0 Redis calls (subsequent)
```

**Bandwidth Savings:** ~83% for authenticated requests
